# MaintainIQ — Architecture (Module 1)

## 1. High-level shape

Monorepo, two independently deployable apps:

```
MaintainIQ/
  client/    React 19 + Vite  → Vercel
  server/    Node.js + Express → Render
```

No shared workspace package — the brief scores FE/BE as separate deploy targets, and a shared-types package adds build complexity with no payoff on plain JS (no TS project refs to share). Duplication between the two `zod`/`express-validator` schemas is small and acceptable.

## 2. Backend layering

```
server/src/
  config/        env loading, db connection, cloudinary, gemini client init
  constants/     roles, asset statuses, issue statuses, allowed transitions
  models/        Mongoose schemas (User, Asset, Issue, MaintenanceRecord, History, Notification)
  validators/    express-validator / zod chains per route
  middlewares/   auth (JWT verify), role-guard, error handler, rate limiter, upload (multer)
  services/      business logic (assetService, issueService, aiService, qrService, cloudinaryService)
  controllers/   thin — parse req, call service, shape response
  routes/        route wiring only
  errors/        custom AppError classes + error codes
  utils/         asset code generator, issue number generator, async handler wrapper
  database/      connection bootstrap, seed script
  app.js         express app assembly (helmet, cors, routes, error handler)
  server.js      http server bootstrap
```

**Why services exist separately from controllers**: the same business logic (e.g. "resolve an issue" — validate maintenance note present, flip issue status, flip asset status, write history, maybe send email) is triggered from multiple controllers/contexts (technician action now, Socket.IO event later, a cron job later). Controllers should never contain business rules directly.

## 3. Key architectural decisions

### 3.1 Status transitions as an explicit state machine
`constants/statusTransitions.js` defines an adjacency map: `{ [currentStatus]: [allowedNextStatuses] }` for both Issue and Asset. `services/issueService.js` checks this map before any status write — this is the single enforcement point for "prevent invalid status transitions," so it can't be bypassed by hitting the API directly (the brief explicitly penalizes frontend-only role/workflow enforcement).

### 3.2 History is append-only, generated by services — never a direct route
There is no `PATCH /history/:id` or `DELETE /history/:id` route at all. Every service method that changes state (create asset, submit issue, assign, start inspection, resolve, reopen, etc.) writes a `History` document as its last step, inside the same function. This guarantees "history cannot be edited manually" by construction rather than by a permission check.

### 3.3 Auth: JWT access token (short-lived, in memory/localStorage) + refresh token (httpOnly cookie)
Access token: 15 min, sent in `Authorization: Bearer`. Refresh token: 7 days, httpOnly + secure cookie, rotated on use, hashed copy stored on the User document so a stolen cookie can be revoked server-side (`POST /auth/logout` clears it). Role is embedded in the access token payload but **every protected route re-checks role against the DB-loaded user**, not just the token claim — closes the gap where an admin is demoted but their still-valid token keeps working.

### 3.4 QR code strategy
QR encodes **only** `{PUBLIC_APP_URL}/assets/public/:assetCode` — never an internal Mongo `_id`, never asset metadata. `assetCode` is a separate immutable field from `name`/`location`, generated once at creation (`utils/assetCodeGenerator.js`, format `AST-XXXXXX`), so renaming an asset never breaks the QR. The public route resolves by `assetCode`, not `_id`. QR image itself is generated server-side with the `qrcode` package and returned as a data URL / PNG buffer — regenerated on demand from the stored URL, never stored as a blob (no drift possible between QR and URL).

### 3.5 Public asset page is a projection, not a permission check
`GET /api/assets/public/:assetCode` uses a dedicated Mongoose `.select()` projection (name, code, category, location, condition, status, lastServiceDate, nextServiceDate, a few safe recent history entries) rather than fetching the full document and filtering in the controller. This makes it structurally impossible for a future field addition (e.g. `internalCost`) to leak onto the public page by accident — someone has to deliberately add it to the projection.

### 3.6 AI service is isolated behind one interface
`services/aiService.js` exports `triageIssue({ asset, complaint, history })`. It owns: prompt construction, calling Gemini, parsing/validating the JSON response against a zod schema, and a fallback path (timeout/invalid JSON/service-down → typed error, not a thrown 500) that the controller turns into a "AI unavailable, enter manually" response. The Gemini API key lives only in `server/.env`, read via `config/env.js`; it is never sent to or readable from the client. AI suggestions are saved on the Issue document with per-field `{ value, aiSuggested, userEdited }` so triage provenance is auditable, satisfying the brief's "store whether a field was AI-suggested and edited."

### 3.7 File evidence upload flow
Multer (memory storage, size/type limits) → Cloudinary upload stream → only the returned secure Cloudinary URL + public_id are persisted on the Issue/MaintenanceRecord document. No file ever touches server disk, which matters for Render's ephemeral filesystem.

### 3.8 Error handling
Single `errors/AppError.js` (statusCode, message, code) thrown from services; `middlewares/errorHandler.js` is the last middleware in `app.js` and is the only place that shapes the JSON error response `{ success: false, message, code }`. Controllers wrap async handlers with `utils/asyncHandler.js` so no route needs a try/catch.

## 4. Frontend shape (preview — detailed in later modules)

```
client/src/
  api/            axios instance + endpoint modules (React Query hooks live next to them)
  components/     shadcn/ui-based reusable components
  features/       auth/, assets/, issues/, maintenance/, dashboard/ — colocated pages+hooks+forms
  routes/         React Router route tree incl. public (no-auth) vs protected layout
  context/         auth context (current user, login/logout)
  lib/            zod schemas mirrored from backend validators, qr helpers
```

TanStack Query owns all server state (cache, refetch, optimistic updates on status changes); Context API only holds the authenticated user + tokens. React Hook Form + Zod for every form, same shape as backend validators to keep client/server validation messages consistent.

## 5. Data flow for the golden-path scenario (from the brief §2)

`Public page → POST /issues (with assetCode)` → `issueService.createIssue` generates issue number, sets asset status → `Issue Reported`, writes History → `AI Issue Triage` (`POST /issues/:id/ai-triage`, technician/admin only, called from the internal dashboard, not the public page) → user edits/accepts → `assignService.assign(issueId, technicianId)` → asset status stays, issue status → `Assigned`, History entry → technician `PATCH /issues/:id/status {Inspection Started}` → asset → `Under Inspection` → technician adds maintenance record + resolves (`PATCH /issues/:id/resolve`, requires a maintenance note to exist — checked in service) → asset → `Operational`, issue → `Resolved`, History entries at every step → optional `POST /issues/:id/ai-summary` for the maintenance summary.

## 6. Deployment topology

- **client** → Vercel (static build, env `VITE_API_URL` pointing at Render backend)
- **server** → Render (Node web service, env vars for Mongo URI, JWT secrets, Cloudinary, Gemini key, allowed CORS origin)
- **database** → MongoDB Atlas (free tier M0, IP allowlist 0.0.0.0/0 or Render's static egress if available)
- **Docker** (bonus): `server/Dockerfile` + `client/Dockerfile` + root `docker-compose.yml` for local dev parity, not required for the actual Render/Vercel deploy
- **GitHub Actions** (bonus): lint + test on PR, optional auto-deploy hook

## 7. What's deliberately NOT included

- No Redis, no message queue, no Socket.IO in the mandatory path — the brief explicitly warns "an incomplete core product with Kafka or Redis is weaker than a complete product with clean engineering." These stay as later bonus additions only if core scope is fully done and stable.
- No multi-tenant/organization model — brief doesn't require it for Track A; a single organization-wide asset pool keeps schema and auth simpler.
